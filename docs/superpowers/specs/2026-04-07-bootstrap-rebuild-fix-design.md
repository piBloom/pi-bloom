# Bootstrap Rebuild Fix — Design Spec

**Date:** 2026-04-07
**Status:** Approved

## Problem

After a fresh NixOS install and running the NixPI bootstrap script, users run:

```
nixos-rebuild switch --flake /srv/nixpi#nixpi
```

This targets the NixPI dev flake directly, which pins `nixpkgs` to `nixos-unstable` (currently resolving to NixOS 26). The host-owned flake at `/etc/nixos/flake.nix` — generated by the bootstrap script with a stable `nixos-25.11` pin and `nixpi.inputs.nixpkgs.follows = "nixpkgs"` — is bypassed entirely. The result is:

1. NixOS upgrades to 26 (unstable), a newer kernel that fails to initialize the GPU on miniPC hardware.
2. Screen goes completely black after reboot — no TTY, no recovery without external boot media.

## Root Causes

| # | Cause | Evidence |
|---|---|---|
| 1 | Wrong rebuild target | `flake.nix:6` pins `nixos-unstable`; `nixosConfigurations.nixpi` alias exposes it as a user-facing rebuild target |
| 2 | Missing GPU firmware | `vps.nix` does not enable redistributable firmware; miniPC GPUs need firmware for KMS to initialize under newer kernels |

## Fix — Four Parts

### Part 1 — `nixpi-rebuild` wrapper script

A new script that wraps the correct rebuild invocation so users never need to type the flake path.

**Files:**
- `core/scripts/nixpi-rebuild.sh` — script body: `nixos-rebuild switch --flake /etc/nixos --impure "$@"` (passes extra args through)
- `core/os/pkgs/nixpi-rebuild/default.nix` — package derivation (same pattern as `nixpi-setup-apply`)
- `flake.nix` — add `nixpi-rebuild` to `packages.${system}`
- `core/os/modules/tooling.nix` — add `nixpi-rebuild` package to `environment.systemPackages`

The script passes `"$@"` so callers can still append flags (e.g. `--dry-run`, `--upgrade`).

### Part 2 — Remove `nixosConfigurations.nixpi` alias

Remove line 122 from `flake.nix`:

```nix
nixosConfigurations.nixpi = self.nixosConfigurations.vps;
```

After this change, `nixos-rebuild switch --flake /srv/nixpi#nixpi` fails with an attribute-not-found error, preventing accidental use. The `nixosConfigurations.vps` config stays; CI and tests reference `vps` directly.

**Checks to update:** the `vps-topology` check currently only verifies `nixosConfigurations.vps` exists — no check requires the `nixpi` alias, so no test changes are needed.

### Part 3 — GPU firmware in `vps.nix`

Add to `core/os/hosts/vps.nix`:

```nix
hardware.enableRedistributableFirmware = lib.mkDefault true;
```

This includes Intel and AMD GPU firmware blobs, ensuring KMS initializes correctly on miniPC hardware. `mkDefault` allows consuming configurations to override it if needed (e.g. a truly minimal cloud VM). The existing comment in `vps.nix` acknowledges monitor-attached x86_64 hosts, so this fits the established intent of the file.

### Part 4 — Post-bootstrap success message

After `nixos-rebuild switch` completes in `nixpi-bootstrap-vps.sh`, add a log message:

```
[nixpi-bootstrap-vps] Bootstrap complete. Use 'nixpi-rebuild' to rebuild or update your system.
```

No functional change — purely UX to close the loop on which command to run going forward.

## Data Flow After Fix

```
Fresh ISO install
      ↓
Run nixpi-bootstrap-vps
      ↓
/etc/nixos/flake.nix generated (nixpkgs = nixos-25.11, nixpi follows nixpkgs)
      ↓
nixos-rebuild switch --flake /etc/nixos --impure  ← bootstrap runs this
      ↓
Reboot → stable NixOS 25.11, GPU firmware present, display works
      ↓
User runs: nixpi-rebuild  ← always correct, alias gone can't go wrong
```

## What This Does Not Change

- The NixPI dev flake (`flake.nix`) continues to use `nixos-unstable` for development and CI.
- `nixosConfigurations.vps` remains for CI/test use.
- The stable release pin (`NIXPI_STABLE_NIXOS_RELEASE=25.11`) in `nixpi-init-host-flake.sh` is unchanged — the bootstrap already generates the correct host flake.

## Testing

The existing `host-flake-bootstrap` check already asserts `! grep -F 'github:NixOS/nixpkgs/nixos-unstable' "$helper"`. No new CI changes needed for Parts 2–4.

Manual test path: fresh NixOS ISO install → run bootstrap → reboot → confirm stable version, working display, `nixpi-rebuild` available.
